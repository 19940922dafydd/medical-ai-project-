

### 1. 医疗专属的立体监控 (Observability)
在医疗问答中，慢一点用户还能忍受，但“胡说八道”是致命的。你需要监控的不仅是基础性能，更是“医疗安全底线”。

* **RAG 召回精准度监控**：医疗知识库通常包含大量的专业指南和药品说明书。你需要监控 Dify 中向量数据库的命中率。比如用户问“二甲双胍饭前吃还是饭后吃？”，你需要看系统是否精准召回了对应的药品说明书分块（Chunk），而不是召回了一篇关于糖尿病前期的科普文章。
* **大模型“拒答率”监控**：在系统提示词中，我们会严格限制大模型：“如果知识库没有相关信息，请直接回答‘我不知道’，绝不能捏造”。你需要监控这个“拒答率”。拒答率过高，说明你的医疗知识库覆盖度不够；拒答率极低但客诉高，说明模型在产生医学幻觉。
* **DeepSeek 消耗与响应 (TTFT)**：医疗急诊或挂号场景下，用户情绪比较焦急。监控模型首字回复时间，如果因为调用外部工具（比如查询排班表 API）导致响应时间超过 3-5 秒，产品上就需要增加类似“正在为您检索医生排班信息…”的过渡状态。

### 2. 医疗 Bad Case 归因与流转
当发现用户对某条回答点了“踩”，或者连续追问了三次“你没懂我意思”，你需要一套标准的归因分析 SOP：

* **口语与术语的鸿沟（意图识别错）**：
    * *Bad Case*：用户问“胸口像压了块石头喘不过气”，Agent 给出了普通哮喘的建议。
    * *归因*：未能识别这是“心肌梗死”的高危非典型描述。
    * *对策*：在工作流的意图分类节点，增加对口语化急症表达的识别。
* **文档切分颗粒度问题（RAG 召回错）**：
    * *Bad Case*：检索到了正确的体检报告解读指南，但因为 Dify 切词（Chunking）时截断了上下文，导致指标对应的单位丢失（比如把 mmol/L 丢了），导致模型判断错误。
    * *对策*：调整分块策略，针对医疗表格或指标，采用按段落或特定医学语义的切分规则。
* **越界诊断（模型幻觉/越权）**：
    * *Bad Case*：Agent 凭借检索到的内容，直接告诉患者“您确诊了XX病，建议购买XX药”。
    * *对策*：强化 Prompt，约束 Agent 只能做“信息提供与导诊”，绝不能下“诊断结论”和“开具处方”。

### 3. 医疗数据飞轮与 Golden Dataset
为了在简历中突出你的严谨性，你需要构建一个“医疗问答黄金测试集”。

* **构建黄金测试集（Golden Dataset）**：收集 100 个极具代表性的医疗 Query。例如：包含多意图的复杂提问（“我昨晚发烧 39度，吃了布洛芬现在退了，还需要去发热门诊吗？”）、药品禁忌查询、化验单数值追问等。
* **回归测试闭环**：未来你在 Dify 中每一次修改系统 Prompt、更换更高版本的 DeepSeek 模型，或者重新向量化了一批新的医学文献，都**必须**先让 Agent 在这 100 个测试集上跑一遍。只有当准确率没有下降，且安全拦截全部生效时，才能发布新版本。
* **动态扩充知识库**：通过分析高频的“未命中”提问，发现最近流感高发，很多患者在问某种新特效药的医保报销比例。迅速将该政策文档补充进知识库，Agent 就能立刻学会。

### 4. 风控兜底与人机协同（核心防御机制）
这部分是医疗 Agent 真正落地的护城河，也是体现产品经理业务深度的关键。

* **前置高危拦截（Router 机制）**：在调用 LLM 之前，先过一层轻量级的意图分类路由。一旦触发“自杀倾向”、“剧烈胸痛”、“大出血”、“孕妇急症”等关键词或语义，直接**熔断**大模型的生成，强制输出预设的硬编码话术：“检测到您可能处于紧急医疗状况，请立即拨打 120 呼叫救护车！”。
* **无缝流转飞书（人机协同）**：当 Agent 遇到无法解析的复杂病情，或者连续两轮对话未能满足患者时，通过 Webhook 触发转人工机制。
    * *产品设计*：Agent 在前端提示患者“正在为您转接专业导诊护士”，同时后台通过 API 将前序的聊天记录、患者画像总结发送到客服团队的**飞书工作群**或飞书服务台。人工客服接管后，不需要患者重复描述病情，直接基于上下文继续服务。
