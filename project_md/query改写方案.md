# 医疗场景 Query 改写 —— 产品化保障方案（准确+可靠）
## 核心产品定位
**Query 改写不是一个NLP算法模块，而是「医疗风控层 + 意图识别增强层」**  
所有设计围绕三句话：

1. 不改错
2. 不改出风险
3. 不改变用户原意

---

# 一、产品化方案1：把改写规则做成「后台可配置词典」，从源头保证准确
## 产品设计动作
1. 做一个 **医疗知识映射后台**（产品功能）
    - 支持：**口语词 ****↔**** 标准医学词** 配置
    - 支持：**错词/简写 → 正确词** 配置
    - 支持：**风险词 → 剥离/过滤** 配置
2. **权限分离**（产品机制）
    - 医学审核人员：只能**新增/审核词条**
    - 产品：配置规则开关、启用/禁用
    - 开发：不参与词条维护，只维护页面
3. **规则只允许“一对一”**（产品约束）
    - 禁止一对多、多对多、模糊匹配
    - 禁止推理式、联想式词条

## 效果
从产品层面杜绝：

+ 开发写死代码导致难维护、难纠错
+ 模型乱编、乱联想医学词汇
+ 改写规则不可控、不可审

## 产品约束（保证可靠）
1. **优先规则，次用模型**  
规则能搞定的，绝对不用模型
2. **只做标准化，不做推理**  
产品层面明确：改写只做「翻译/清洗/规整」
3. **复杂问句只拆词，不理解逻辑**

---

# 三、产品化方案3：内置「三重自动校验器」产品功能，拦截错误改写
这是**产品必须设计的质控功能**，不需要算法复杂能力：

1. **信息完整性校验**  
改写后必须保留原句**所有医疗关键词**，丢词则放弃改写
2. **原意不变校验**  
产品规则：改写结果不能扩大、缩小、扭曲用户问题
3. **合规性校验**  
改写后**不允许出现**：诊断、治疗、用药、风险判断词汇

**产品逻辑：**  
任一校验不通过 → **自动熔断改写，直接使用原Query**

---

# 四、产品化方案4：建立「改写测试集」产品化准入机制
## 产品设计
1. 后台内置 **医疗Query 测试用例库**（500～1000条）  
覆盖：口语、错字、倒装、长句、风险句
2. 增加 **改写效果仪表盘**（产品看板）
    - 改写准确率
    - 合规率（必须=100%）
    - 意图识别提升率
3. **上线准入规则**（产品机制）
    - 准确率 < 95% 不准上线
    - 出现1条风险改写 → 回滚规则

这就是**产品化质量门禁**。

# 五、产品化方案5：前端+后端「产品化反馈入口」，线上纠错
## 前端产品（用户可见）
+ 回答下方加：  
**【回答不相关】【回答不准确】**  
点击 → 自动上报：原Query + 改写结果 + 回答

## 后端产品（运营可见）
+ 自动归集 **改写错误Top 20**
+ 医学审核人员 → 直接在后台修正词条
+ 支持**一键生效、一键回滚**（产品功能）

---

# 六、产品化方案6：医疗场景专属「改写熔断兜底」
这是**医疗产品必须有的安全设计**：

产品层面明确 4 种熔断条件，满足任意一条就**不改写**：

1. 无匹配规则 → 不改
2. 模型置信度低 → 不改
3. 多症状、多条件复杂句 → 只拆词，不深度改写
4. 歧义模糊问题 → 直接用原句

**产品原则：宁可不改，绝不乱改。**

---

# 七、产品化方案7：建立「医疗改写迭代闭环」
产品设计一套固定运营流程：

```plain
用户query → 改写 → 校验 → 分流 → 反馈 → 审核 → 更新词典 → 测试 → 上线
```

+ 每周更新一次词典
+ 每月全量回归测试
+ 所有改动留痕、可追溯

# 面试口述 1 分钟精简版（直接背）
我会通过**一套完整产品化方案**，确保医疗场景Query改写的准确性和可靠性：

第五，设计**前后端反馈入口**，线上错误实时归集修正；  
最后设置**熔断兜底机制**，宁可不改也不乱改。

这套产品化方案，既保证改写精准，又从产品机制上守住医疗安全红线，真正可上线、可落地。
