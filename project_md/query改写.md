# 一、医疗场景为什么不改写，一定会出大问题？
用户真实提问全是**口语、模糊、省略、倒装**，直接丢给意图识别/关键词匹配，10 个错 6 个：

| 用户原始口语Query（不改写） | 机器理解困难 | 改写后标准Query | 意图识别结果 |
| --- | --- | --- | --- |
| 孕晚期肚子硬邦邦的 | 匹配不到关键词 | 孕晚期 假性宫缩 正常表现 | 孕期护理（热门） |
| 高血压能不能吃咸的 | 无标准词 | 高血压患者 低盐饮食 要求 | 慢病护理（热门） |
| 体检那个转氨酶高咋回事 | 模糊 | 体检 转氨酶偏高 指标解读 | 报告解读 |
| 我这头疼是不是大病 | 危险+模糊 | 头痛 常见原因 科普 | 症状科普 |


不改写 = 意图识别失效 = 要么答不出，要么乱答 = 医疗场景**致命体验问题**。

# 二、医疗 Query 改写的 3 条**铁律**（绝对不能碰）
你在面试里说这 3 条，直接体现你**懂医疗、懂安全、懂落地**：

1. **只做“翻译”，不做“推理”**  
只把口语→标准词，**绝不替用户判断疾病、风险、病因**。
2. **只做“补全”，不做“新增”**  
只补关键词，不增加医学信息。
3. **只做“简化”，不做“联想”**  
越简单越安全，拒绝复杂思维链改写。

---

# 三、我们只做这 4 类改写（80% 问题全覆盖，零风险）
### 1. 口语 → 标准医学词（最核心）
+ 肚子硬 → 假性宫缩
+ 嗓子疼 → 咽痛
+ 吃咸的 → 高盐饮食
+ 血糖高了 → 血糖异常

### 2. 省略/模糊 → 补全关键词
+ 胎动多少 → 孕晚期 胎动 正常范围
+ 挂什么科 → 腹痛 科室选择

### 3. 长句复杂 → 拆成核心词
+ 孕38周又宫缩又见红怎么办  
→ 孕晚期 宫缩 见红 科普

### 4. 错别字/口语简写 → 修正
+ 孕产捡 → 产检
+ 糖鸟病 → 糖尿病

# 四、绝对不做的改写（医疗红线）
+ ❌ 不做：用户说“不舒服”→ 改写为“可能是感冒”（推理病情）
+ ❌ 不做：用户说“头疼”→ 改写为“偏头痛如何治疗”（下诊断）
+ ❌ 不做：多轮复杂 Cot、HyDE 改写（容易编内容）

# 六、落地技术方案（产品面试可直接讲，极低成本）
你不用讲复杂算法，就讲这套**「规则优先+轻量小模型兜底」**：

## 1）80% 问题：规则改写（关键词映射表）
你作为产品经理，直接输出一张表：  
`口语词 → 标准医疗词`  
交给开发写正则就能跑，**不用大模型、零成本**。

## 2）20% 复杂问题：轻量小模型改写
用 1B～3B 小模型（本地可跑），只干一件事：  
**把口语变成标准关键词组合**。

Prompt 模板（你可以直接用）：

```plain

输出格式：关键词1 关键词2 关键词3

用户问题：{query}
标准关键词：
```

# 七、面试完整口述话术（1 分钟，直接背）
在我们医疗问答的意图识别环节，**我会设计一层轻量的 Query 改写**，它是意图识别的前置增强。

因为用户提问都是口语化、模糊化的，直接识别会匹配失败、理解错误。  
但医疗场景我只做**安全可控的标准化改写**：把口语转成标准医学词、把模糊问题补全关键词，**绝不做病情推理和内容新增**。

改写后再进入意图识别，既能保证意图匹配精准，又守住医疗安全红线，  
再结合我们之前的热门/小众分流，既准确又低成本。
